# 🗺️ 第二部分：标准工作流 (从需求到上线)

## 4. 火炬 UI 设计全流程{#4.火炬 UI 设计全流程}

### 4.1 需求评审：如何看懂PRD与反讲需求{#4.1需求评审：如何看懂PRD与反讲需求}
1. **接收需求**：产品下发PRD（产品需求文档，在企业微信-微盘-技术部文件里查找），明确产品类型、客户需求、核心功能、主要风格、项目优先级。
2. **需求对齐**：主动与产品沟通，确认需求细节（避免理解偏差），如有不合理需求应及时与产品沟通。
3. **制定设计计划与工时参考**：
    - **🟢 换皮/复用类**：仅替换切图与色值，**预计 1 - 1.5 个工作日**。
    - **🟡 局部新增类**：涉及1-2个新增复杂弹窗或游戏面板，**预计 2 - 5 个工作日**。
    - **🔴 全案定制类**：涉及一二三级页面全设计，**需提前报备组长和主管，预留 10 个工作日以上**（含至少3轮评审）。

### 4.2 设计执行：自查清单（空态/极限值/权限）{#4.2设计执行：自查清单（空态/极限值/权限）}
1. **高保真设计**：项目应保证美观、切图完整、质量达标、效率高。
2. **自查清单**：
    - **无障碍**：对比度是否足够？字号是否清晰？
    - **组件状态**：空态、加载态、错误态是否已设计？（**列表为空只给白板属于不合格交付**）
    - **交互规范**：点击热区是否不小于44x44px？
    - **极限情况**：昵称最长时如何截断？金额数字过大时如何显示？
3. **游戏/动效设计**：蓝湖中有很多游戏的种类，比如转盘、抽奖、押宝等，不同的游戏有不同的玩法，不过大致都分为首页、排行榜、规则、奖池、榜单等几个主要页面，动效一般需要设计游戏背景动效（也叫启屏动效，与游戏高度一致）和大奖动效（全屏高度，抽到价值较高礼物的时候展示，没有大奖玩法的则不需要设计，需要配套音效）

### 4.3 交付物标准：切图命名规范表与图层整理要求{#4.3交付物标准：切图命名规范表与图层整理要求}
1. **评审与迭代**：
    - **产品评审**：在蓝湖中与产品经理核对产品功能清单上的内容，确保功能完整、不额外增加功能，不缺少必要功能。
    - **客户评审**：下载蓝湖组合图并发送到对接群，艾特客户审查，清晰表达设计思路，认真记录评审意见（区分必改项、可选项、延后项），不纠结“审美分歧”，重点关注落地性和用户体验。
    - **方案迭代**：根据评审意见修改设计稿，修改完成后同步导师/产品确认，直至通过。
2. **交付物**：
    - 核对最终设计稿、标注文件、切图。
    - 若使用 Photoshop 设计，需特别注意确认切图位置（PS对蓝湖插件兼容性一般）。
    - 若有游戏动效设计，则需要导出为svga格式文件（存好备用），然后转换为vap格式文件（尾缀为.mp4是正常的，因为vap格式是给.mp4的视频右侧加一个alpha蒙版），将转换好的vap以及给动效配的音效（.mp3文件）打成压缩包发给开发人员。

### 4.4 走查与验收：设计还原度打分表{#4.4.走查与验收：设计还原度打分表}
1. **开发对接**：解答开发人员的疑问，协助开发还原设计效果，避免因沟通偏差导致还原度不足。
2. **走查验收**：开发完成后，将测试包下载到测试机中，对照设计稿进行走查，记录还原度问题，提交给测试人员或直接告知前端，跟进修复直至验收通过。

## 5. 上线后复盘与沉淀{#5.上线后复盘与沉淀}
- **经验沉淀**：总结本次需求的问题（如需求理解偏差、评审效率低），避免后续重复踩坑。
- **迭代优化**：根据用户反馈，与产品和开发人员同步后优化高保真设计稿，进行下一轮迭代需求。
