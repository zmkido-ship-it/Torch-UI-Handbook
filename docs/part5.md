# 第五部分：生存法则与红线🛡️

## 设计评审话术模板

- **少说**：“我觉得不好看。”
- **多说**：“从用户角度，这个设计可能导致XX操作不便；从开发角度，这样调整能避免增加额外工作量。”

## 与前端开发的“互译词典”

- **开发说“无法实现”时**：先问清是逻辑限制？还是成本过高？亦或是需求对接问题？提前与产品和开发人员共同探讨需求文档，不硬刚，不盲从。
- **尊重技术边界**：设计方案需考虑技术可行性、开发成本，避免无效设计。
- **文件管理**：设计稿按规范命名、归档，及时更新版本，避免出现“旧版误用”；需协同设计的文件应共享给相关人员并设置对应权限。

## 职业素养与持续学习

- **产品至上**：所有设计围绕“用户体验”和“产品需求文档”，不盲目追求视觉美观，不额外添加功能，不缺少必要功能。
- **务实落地**：注重开发的可实现程度。
- **细节控**：杜绝低级错误。
- **持续学习**：主动关注设计趋势、行业案例，定期总结沉淀。
